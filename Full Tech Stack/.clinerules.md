# .clinerules

## Neeva AI v1.0 — Project Rules

### Project Identity

Neeva AI is an AI-powered mobile-first wellness platform.

The objective is **not** to redesign the application.

The objective is to transform the existing web implementation into a production-grade React Native + Expo application while preserving the product experience.

---

# Core Principle

> Preserve the product. Replace the rendering engine.

Do NOT rebuild Neeva.

Do NOT redesign Neeva.

Replace only the web rendering layer with React Native.

---

# Product Freeze

The following are permanently locked for v1.

## Product

* AI Wellness Platform
* Mobile First
* Wellness Journey First
* User Outcomes Over Features

## Navigation

* Home
* Chat
* Journey
* Community
* Profile

No navigation changes unless explicitly requested.

---

# UI Freeze

Neeva Design System v1.0 is frozen.

Do NOT:

* redesign layouts
* change colors
* change typography
* replace spacing
* change component hierarchy
* introduce Material Design

Only implement the approved design.

# Repository Rules

Transform the existing repository.

Do NOT create a new long-term repository.

Preserve:

* Git history
* Documentation
* Backend
* API
* Product structure

---

# Migration Rules

Preserve:

* Business logic
* Prompt system
* AI services
* Types
* Validation
* Query hooks
* Zustand stores
* Backend APIs

Replace:

* DOM components
* Web UI components
* Web rendering layer
* Web navigation

---

# Architecture Rules

Follow Feature-First Architecture.

Preferred structure:

src/

* core/
* features/
* shared/
* services/
* theme/
* constants/
* config/
* hooks/
* types/

Each feature owns:

* UI
* Hooks
* Repository
* Types
* Tests

Avoid technology-first folders.

---

# Component Rules

Build reusable components.

Never duplicate UI.

Examples:

* Button
* GlassCard
* Avatar
* ProgressRing
* JourneyCard
* ChatBubble
* Input
* Modal
* BottomSheet
* Toast
* Navigation

Every screen should compose reusable components.

---

# Design Tokens

Never hardcode design values.

Use centralized tokens.

Examples:

* colors
* spacing
* typography
* radius
* shadows
* gradients
* motion
* zIndex

---

# Data Flow

Always follow:

Screen

↓

Feature Hook

↓

Repository

↓

API Client

↓

Backend

Screens must never communicate directly with external services.

---

# AI Rules

Never call NVIDIA directly from UI.

Always use:

UI

↓

AI Service

↓

Provider Adapter

↓

NVIDIA NIM

Future AI providers must be replaceable without changing UI code.

---

# Engineering Rules

Prefer:

* Composition over duplication
* Reusable components
* Small focused files
* Type safety
* Clear naming

Avoid:

* Massive components
* Business logic inside UI
* Duplicate code
* Deep prop drilling
* Hardcoded constants

---

# Performance

Prefer:

* Lazy loading
* Memoization where appropriate
* FlatList for lists
* Image optimization
* Efficient animations

Do not optimize prematurely.

---

# Security

Never expose:

* API keys
* Secrets
* Tokens

Always validate:

* Input
* Authentication
* Authorization

Use Secure Store for sensitive local data.

---

# Accessibility

Every component should support:

* Minimum 44x44 touch targets
* Proper accessibility labels
* Screen readers
* Dynamic font scaling
* Sufficient color contrast

---

# Analytics

Track only meaningful product events.

Examples:

* Mood Check-in
* Journey Started
* Journey Completed
* Chat Started
* Meditation Completed
* Reflection Saved
* Subscription Purchased

Do not add unnecessary analytics events.

---

# Coding Standards

* TypeScript strict mode.
* No any unless unavoidable.
* Functional components only.
* Hooks before helpers.
* Named exports preferred.
* Clear file naming.
* Keep functions focused.

---

# Development Philosophy

Do not introduce new features during migration.

Migration goals:

1. Preserve functionality.
2. Improve architecture.
3. Improve maintainability.
4. Achieve native performance.
5. Reach feature parity.

Feature development begins only after migration is complete.

---

# Decision Policy

If multiple implementation options exist:

1. Preserve the current product behavior.
2. Prefer the simpler architecture.
3. Minimize technical debt.
4. Keep business logic reusable.
5. Ask before making architectural changes.

Never make irreversible architectural decisions without confirmation.

---

# Definition of Success

The migration is successful when:

* Users experience the same Neeva product.
* Native performance replaces web performance.
* Backend remains compatible.
* Business logic is preserved.
* UI matches the approved design system.
* Codebase is modular, scalable, and maintainable.

The goal is not to create a different app.

The goal is to make the existing product feel like it was built natively from day one.
